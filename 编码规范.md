# 你在所有编码任务中必须严格遵循以下规则

## 第一部分：设计原则与设计模式（如何使用）

1. 第一性原理拆解
    - 复杂需求先拆成小任务。
    - 每次只实现一个小任务，完成后先验证再继续。
2. 设计原则（KISS / SOLID / 单一职责）
    - 设计规划遵循第一性原理；代码实现遵循 KISS 与 SOLID。
    - 代码修改遵循单一职责，不混合多个变更。
    - 优先复用现有抽象，避免重复代码。
    - 确保代码可读性与可维护性，必要时加简要注释。
3. 常用设计原则
    - 迪米特法则：只与直接依赖交互，避免跨层链式调用。
    - 开闭原则：对扩展开放、对修改关闭。
    - 依赖倒置原则：依赖抽象，不依赖具体实现。
    - 接口隔离原则：接口保持小而专一。
    - 组合优于继承：优先组合复用，减少耦合。
    - 关注点分离：业务、数据、基础设施分层实现。
4. 设计模式使用规范
    - 设计模式按需使用，不为模式而模式。
    - 采用设计模式时，必须与现有架构分层、命名、风格与错误处理约定保持一致。
    - 未被要求时，保持接口与数据契约向后兼容。

## 第二部分：日常开发规范（执行流程）
   
1. 最小改动原则
   - 变更范围最小化，控制影响面，避免大范围修改。
   - 不增加无关逻辑/文件/配置/依赖。
   - 未明确要求时，不改变既有行为。
   - 在已有功能基础上添加新功能时，必须确保：
     - 不影响原有功能
     - 不添加其他功能、代码、逻辑、文件、配置、依赖
2. 架构一致性
   - 遵循现有架构设计，保持分层、命名、风格与错误处理约定一致。
3. 质量闸门
   - 修改前：识别影响路径与调用链。
   - 每个子任务完成后：执行相关 compile/lint/test 验证。
   - 修改后：做回归、边界和逻辑自检，确保无遗漏。
4. 沟通与阻塞处理
   - 按顺序汇报：改了什么、为什么、如何验证、剩余风险。
   - 被阻塞或需求不清时，先提问确认，不得猜测实现。
